# 系統修改規格書：料槍維修條碼化輸入與權限控管變更

## 1. 變更概述 (Change Overview)

本案變更旨在優化現行「料槍維修」作業流程。

* **痛點解決**：過去人員需以目視尋找並手動選取/填寫料槍編號，易造成人為防呆破口（選錯料槍）。
* **優化方案**：於主頁新增「掃碼維修」功能，強制透過 Barcode Sensor（條碼掃描槍）刷入料槍編號，自動帶出防錯資料，提升作業效率與準確度。
* **安全性變更**：針對主頁查詢清單中的特殊維修按鈕，實施基於登入帳號的特定人員權限控管。

---

## 2. 系統架構與關聯 (System Architecture & Dependency)

* **前端技術**：ASP.NET MVC Razor View + SweetAlert2 (提示視窗)
* **後端技術**：.NET C# + Dapper / EF Core
* **資料庫**：MSSQL 

---

## 3. 功能需求規格 (Functional Requirements)

### 3.1 【功能一】料槍掃碼維修作業

#### A. 進入點與命名

* **位置**：料槍維修主頁（`/MaterialGun/MaterialGunRepairView`）上方工具列。
* **按鈕命名**：`[ 掃碼維修 ]` (Scan & Repair) — *比「料槍維修」更具行為引導性，避免與原主頁名稱混淆。*

#### B. 掃碼維修頁面 (`/MaterialGun/MaterialGunScanRepair.cshtml`) 流程與 UI 行為

1. **初始狀態**：
* 頁面載入後，系統自動將 Focus（焦點）鎖定在 `料槍條碼輸入框 (txtGunBarcode)`。
* 其餘編輯欄位（依照`/MaterialGun/MaterialGunRepair?sno=1760`介面，如：檢修不良分類、檢修不良原因、更換部品名稱、更換零件、更換部品料號）皆為 **Disabled (唯讀)** 強制人員必須先掃碼。


2. **掃碼輸入 (Barcode Sensor 觸發)**：
* 當人員使用掃描槍刷入料槍編號後（通常帶有 `Enter` 鍵值 `Keycode 13`），Vue.js 監聽事件並觸發 Ajax API 請求。


3. **後端驗證與資料帶出**：
* API 檢查該料槍編號是否存在且狀態是否允許維修。
* **成功**：解鎖編輯欄位，自動帶出該料槍的「依照`/MaterialGun/MaterialGunRepair?sno=1760`介面，如：檢修不良分類、檢修不良原因、更換部品名稱、更換零件、更換部品料號」」等基礎資料。
* **失敗**：使用 `SweetAlert2` 彈出錯誤提示（如：「查無此料槍編號」或「該料槍目前已完成維修，無法維修！」），清空輸入框並重新 Focus。


4. **存檔與返回**：
* 人員輸入完成，點擊 `[ 儲存維修紀錄 ]`。
* 後端寫入資料庫成功後，前端觸發 `Swal.fire` 成功綠色勾勾提示。
* 點擊提示框的「確定」或 1.5 秒後，**自動跳轉/返回料槍維修主頁**。


### 3.2 【功能二】特定人員維修功能權限控管 (已完成維修之資料)

#### A. 控制對象

* **位置**：料槍維修主頁的查詢結果清單（Grid/Table）。
* **控管按鈕**：清單中針對「料槍維修清單」資料列中`Chk`(是否完修(data-chk="False"=>已完修 Or data-chk="True"=>未完修"))為0時，所綁定的 `[ 料槍維修 ]` 功能按鈕。

#### B. 權限判定邏輯

* **後端驗證（主要資安防線）**：
* 在 Controller / API 層級，直接抓取當前 Session 或 JWT 中的登入工號/帳號（例如 `User.Identity.Name`）。
* 與系統設定的「允許名單（白名單-登入帳號`02898`）」進行比對（直接於`appsettings.json`的特權清單）。


* **前端呈現（優化 UX）**：
* 若目前登入帳號**不在**指定名單內，保持原邏輯判斷，不做任何更動。
