## 1. 專案背景與目的

### 1.1 背景說明

昶亨整合式系統（ESIntegrateSys）中的「料槍維修」模組，目前供人員對注料槍的維修作業進行記錄與管理。現行作業流程存在人為選錯料槍的防呆破口，且部分敏感維修功能缺乏細粒度的人員權限管控。

### 1.2 變更目的

本次系統修改旨在達成以下兩個目標：

1. **提升作業效率與準確度**：導入條碼掃描（Barcode Sensor）機制，強制透過掃碼方式輸入料槍編號，取代原本的目視選取，消除人為選錯料槍的風險。
2. **強化資訊安全**：針對料槍維修主頁清單中，已完修資料列所綁定的特殊維修按鈕，實施基於登入帳號白名單的特定人員存取控制。

| # | 問題描述 | 影響層面 | 嚴重程度 |
|---|---------|---------|---------|

## 4. 使用者角色與權限

### 4.1 角色定義

| 角色代號 | 角色名稱 | 說明 |
|---------|---------|-----|
| ROLE_OPERATOR | 一般操作人員 | 可執行新增掃碼維修，但不可操作已完修資料的維修按鈕 |
| ROLE_PRIVILEGED | 特權維修人員 | 除一般操作員權限外，可操作已完修資料的維修按鈕（帳號：`02898`） |
| ROLE_VIEWER | 唯讀人員 | 僅能查詢維修清單，無法執行任何寫入操作（視現有系統設定而定） |

### 4.2 功能權限矩陣

| 功能 | 一般操作人員 | 特權維修人員 | 唯讀人員 |
|-----|------------|------------|---------|
| 查詢料槍維修清單 | 可 | 可 | 可 |
| 使用掃碼維修（新增維修記錄） | 可 | 可 | 不可 |
| 操作未完修料槍之 `[ 料槍維修 ]` 按鈕（`data-chk="True"`） | 可（依現有邏輯） | 可 | 不可 |
| 操作已完修料槍之 `[ 料槍維修 ]` 按鈕（`data-chk="False"`） | **不可** | **可** | 不可 |

## 5. 功能需求

### 5.1 【F01】料槍掃碼維修作業

#### 5.1.1 進入點

- **頁面**：料槍維修主頁（路由：`/MaterialGun/MaterialGunRepairView`）
- **操作**：於主頁上方工具列新增 `[ 掃碼維修 ]` 按鈕
- **按鈕命名依據**：命名為「掃碼維修」而非「料槍維修」，具行為引導性，且避免與原主頁名稱混淆

#### 5.1.2 掃碼維修頁面

- **頁面路徑**：`/MaterialGun/ScanRepair.cshtml`
- **對應路由**：`/MaterialGun/ScanRepair`

#### 5.1.3 頁面初始狀態

- 頁面載入完畢後，系統自動將輸入焦點（Focus）鎖定於 `料槍條碼輸入框（txtGunBarcode）`
- 以下欄位初始狀態為 **Disabled（唯讀）**，強制人員先完成掃碼後才可編輯：
  - 檢修不良分類
  - 檢修不良原因
  - 更換部品名稱
  - 更換零件
  - 更換部品料號

#### 5.1.4 掃碼輸入事件流程

1. 人員使用條碼掃描槍刷入料槍條碼
2. 掃描槍自動送出 `Enter` 鍵（`KeyCode: 13`）
3. Vue.js 監聽 `keydown` 或 `keyup` 事件，偵測到 `Enter` 後觸發 Ajax API 呼叫

#### 5.1.5 後端驗證邏輯

| 情況 | 後端行為 | 前端回應 |
|-----|---------|---------|
| 料槍編號存在且狀態允許維修 | 回傳料槍基礎資料（200 OK） | 解鎖編輯欄位，自動填入資料 |
| 查無此料槍編號 | 回傳錯誤訊息（400 或 404） | SweetAlert2 彈窗顯示「查無此料槍編號」，清空輸入框並重新 Focus |
| 料槍已完成維修（無法再次維修） | 回傳錯誤訊息（400） | SweetAlert2 彈窗顯示「該料槍目前已完成維修，無法維修！」，清空輸入框並重新 Focus |

#### 5.1.6 存檔與返回流程

1. 人員填寫完欄位後，點擊 `[ 儲存維修紀錄 ]`
2. 後端寫入資料庫成功
3. 前端觸發 `Swal.fire` 成功綠色勾勾提示訊息
4. 人員點擊提示框「確定」按鈕，**或** 1.5 秒倒數後自動跳轉，返回料槍維修主頁（`/MaterialGun/MaterialGunRepairView`）

### 5.2 【F02】特定人員維修功能權限控管

#### 5.2.1 控管對象

- **頁面**：料槍維修主頁（`/MaterialGun/MaterialGunRepairView`）的查詢結果清單（Grid/Table）
- **控管按鈕**：清單資料列中，`data-chk="False"`（已完修）的 `[ 料槍維修 ]` 功能按鈕

#### 5.2.2 後端權限驗證（主要資安防線）

- 在 Controller / API 層，讀取當前 Session 或 JWT 中的登入帳號（`User.Identity.Name`）
- 與 `appsettings.json` 中設定的特權帳號白名單進行比對
- 若帳號**不在**白名單內，回傳 `403 Forbidden`，並禁止執行維修操作

#### 5.2.3 前端 UX 呈現

- 若當前登入帳號**不在**指定名單內：保持現有邏輯，已完修按鈕不改變外觀（依需求說明不做額外 UI 變更）
- 若當前登入帳號**在**白名單內：按鈕可正常點擊並進入維修頁面

> **注意**：前端呈現為 UX 輔助，後端驗證為主要資安防線，兩者須同時實施。

#### 5.2.4 白名單設定

- **設定位置**：`appsettings.json`
- **設定範例**：

```json
{
  "PrivilegedRepairAccounts": ["02898"]
}
```

## 6. 非功能需求

| 項目 | 需求描述 |
|-----|---------|
| 效能 | 掃碼觸發後，後端 API 應於 2 秒內完成驗證與資料回傳 |
| 安全性 | 後端權限驗證為強制執行，不可僅依賴前端控制 |
| 相容性 | 前端需相容主流瀏覽器（Chrome、Edge），支援 Barcode Sensor 硬體輸入模擬 Enter 鍵 |
| 可維護性 | 白名單帳號透過 `appsettings.json` 設定，無需修改程式碼即可調整名單 |
| 使用者體驗 | 掃碼後頁面自動帶出資料、自動 Focus、自動跳轉，減少人員手動操作步驟 |

## 7. 畫面流程 / 操作流程

### 7.1 【F01】掃碼維修操作流程

```
[料槍維修主頁]
      |
      | 點擊 [ 掃碼維修 ] 按鈕

      | Enter (KeyCode 13) 觸發
      v

      |
      |--- [驗證失敗] ---> SweetAlert2 錯誤提示 --> 清空輸入框 --> 重新 Focus (回到掃碼步驟)
      |
      |--- [驗證成功] ---> 解鎖編輯欄位，自動帶出料槍資料
                              |

                              |

                              |

                              |

                              |
                        [返回料槍維修主頁]
```

## 8. 資料結構 / API

### 8.1 API 清單

| API 編號 | 方法 | 路由 | 說明 |
|---------|------|-----|-----|
| API-01 | GET | `/MaterialGun/GetGunByBarcode?barcode={code}` | 依條碼查詢料槍資訊及允許維修狀態 |
| API-02 | POST | `/MaterialGun/SaveScanRepair` | 儲存掃碼維修紀錄 |

### 8.2 API-01 回應 DTO（GetGunByBarcode）

```csharp
/// <summary>掃碼查詢料槍資料回應 DTO</summary>
public class GunBarcodeResultDto
{
    /// <summary>料槍流水號</summary>
    public int Sno { get; set; }
    
    /// <summary>料槍編號</summary>
    public string GunCode { get; set; }
    
    /// <summary>檢修不良分類</summary>
    public string RepairCategory { get; set; }
    
    /// <summary>檢修不良原因</summary>
    public string RepairReason { get; set; }
    
    /// <summary>更換部品名稱</summary>
    public string PartName { get; set; }
    
    /// <summary>更換零件</summary>
    public string ReplacePart { get; set; }
    
    /// <summary>更換部品料號</summary>
    public string PartNo { get; set; }
    
    /// <summary>是否允許維修（false = 已完修，不允許）</summary>
    public bool IsRepairAllowed { get; set; }
    
    /// <summary>錯誤訊息（驗證失敗時使用）</summary>
    public string ErrorMessage { get; set; }
}
```

### 8.3 API-02 請求 DTO（SaveScanRepair）

| 驗收項目 | 驗收條件 | 測試方法 |

## 10. 風險與待確認事項

### 10.1 待確認問題

| 編號 | 待確認項目 | 影響範圍 | 需由誰確認 |
|-----|---------|---------|---------|
| Q01 | 已完修料槍的 `[ 料槍維修 ]` 按鈕，若非白名單帳號，前端是否需要 **隱藏** 或 **Disabled** 顯示？或維持現狀僅後端擋？ | F02 前端 UX | 業務/主管 |
| Q02 | `appsettings.json` 白名單日後是否需要做成後台 UI 動態管理（而非每次修改設定檔並重啟）？ | F02 可維護性 | 業務/主管 |
| Q03 | 掃碼維修頁面是否需要沿用原 `MaterialGunRepair?sno=1760` 的完整欄位結構，還是僅需核心欄位？ | F01 資料結構 | 系統分析師 + 業務 |
| Q04 | 料槍條碼的格式與長度是否有規範？是否需要正規表達式前端驗證？ | F01 資料驗證 | 現場 / 硬體規格 |

### 10.2 已知風險

| 編號 | 風險描述 | 風險等級 | 因應措施 |
|-----|---------|---------|---------|
| R01 | 前端若僅依賴 KeyCode 13 判斷掃碼結束，部分掃描槍可能使用不同結尾符號（Tab 或無結尾） | 中 | 確認現場掃描槍型號與設定，必要時改為固定長度自動觸發 |

| R03 | 若瀏覽器 Focus 在其他欄位，掃碼輸入可能進入錯誤欄位 | 中 | 加入頁面 Blur 事件或點擊空白處自動返回 Focus 機制 |
