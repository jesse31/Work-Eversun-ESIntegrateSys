# MVP 階段 0：定義 MVP 目標與「不做什麼」
## 料槍維修條碼化輸入與權限控管變更

**文件日期**：2026-06-04
**需求來源**：`20260603-料槍維修條碼化輸入與權限控管變更.md`
**分析者**：系統架構師（AI 輔助）

---

## 1. MVP 目標

> **以最少的改動，解決料槍維修作業中「選錯料槍」的人為失誤問題，並鎖定特定人員才能對已完修資料進行再次編輯。**

### 成功標準（可驗收）

| 成功標準 | 驗收條件 |
|---------|---------|
| 條碼掃描輸入可用 | 操作人員拿起條碼槍掃描，料槍資訊自動帶出，不需手動輸入編號 |
| 防錯機制生效 | 掃到無效 / 已完修料槍時，SweetAlert2 彈出明確錯誤提示，輸入框清空並重新聚焦 |
| 權限控管生效 | 白名單帳號（02898）以外的使用者，無法操作「已完修」料槍的維修按鈕 |
| 系統穩定 | 兩項功能上線後，現有料槍維修主頁、查詢清單、維修紀錄功能不受影響 |

---

## 2. 核心功能清單（MVP 必須完成）

| 項次 | 功能 | 說明 | 預估難度 |
|-----|------|------|---------|
| F1 | 掃碼維修頁面（ScanRepair） | 新增 `/MaterialGun/ScanRepair` 頁面，初始 Focus 鎖定在條碼輸入框，其餘欄位 Disabled | 中 |
| F2 | 條碼掃入驗證 API | 後端 API 驗證料槍編號存在性與維修狀態，回傳允許/拒絕與基礎資料 | 中 |
| F3 | 欄位自動帶出與解鎖 | 掃碼成功後，前端自動填入料槍相關欄位並解鎖編輯 | 中 |
| F4 | SweetAlert2 錯誤提示 | 查無料槍、已完修等情境的標準化彈窗提示 | 低 |
| F5 | 儲存維修記錄 | 點擊儲存後寫入 DB，成功後自動跳轉回維修主頁 | 低 |
| F6 | 主頁「掃碼維修」入口按鈕 | 在 `/MaterialGun/MaterialGunRepairView` 上方工具列新增按鈕 | 低 |
| F7 | 後端白名單權限驗證 | Controller/Action 層比對 Session 工號與 appsettings 白名單，非白名單者 403 Forbidden | 中 |
| F8 | 前端已完修按鈕條件控制 | 依登入帳號判斷，非白名單者已完修資料的「料槍維修」按鈕不可操作（保持現有 UX 邏輯） | 低 |

**總計**：8 個功能點，預估 **5～8 個工作天** 可交付。

---

## 3. 排除功能清單（MVP 絕對不做）

| 排除項目 | 排除原因 |
|---------|---------|
| 動態白名單管理頁面（後台設定） | 白名單初期僅 1 人（02898），維護複雜度遠超需求，appsettings 足夠；MVP 後若需求擴大再建後台 |
| 行動裝置 App / PWA | 現場使用 USB/藍牙條碼槍配合 PC 瀏覽器，不需原生 App |
| 即時通知推播（SignalR / WebSocket） | 維修狀態目前無跨人員即時協作需求，輪詢或頁面重整即可 |
| 維修流程的完整狀態機（多階段審核） | 現有流程已有 `Chk` 欄位控制，MVP 不重新設計流程 |
| 批次掃碼（一次掃多支料槍） | 核心痛點是「選錯料槍」，非「效率不足」，一次一筆已足夠驗證 |
| 歷史維修數據圖表 / 報表 | 資料分析非本次變更目的 |
| 自動列印維修工單 | 現有流程無此需求 |
| 多語系支援 | 系統已全繁體中文，無多語需求 |
| 密碼安全性重構（SHA-256）| 重要但與本次需求無直接關聯，另立 Issue 處理 |
| Serilog 引入 | 重要但為全系統改動，不在本次 MVP 範圍 |

## 5. 技術債防範

### 5.1 高風險做法（看似簡單但會留技術債）

| 風險做法 | 問題說明 | 建議替代方案 |
|---------|---------|------------|
| 白名單硬編碼在 Controller | 程式碼內寫死 `userId == "02898"`，日後擴充或修改需重新部署 | 必須讀取 `appsettings` / 設定檔，不得硬編碼在程式碼中 |
| 僅做前端按鈕隱藏（無後端驗證） | 使用者可直接呼叫 API 繞過前端限制，資安破口 | 後端 Controller/Action 必須獨立驗證權限（前後端雙重防線） |
| 掃碼驗證 API 未加 Try-Catch | 未捕獲的例外直接回傳 500，使用者看到白頁 | 所有 API 必須有 try-catch，回傳統一的 `{ success, message, data }` 格式 |
| 直接用字串拼接 SQL | `WHERE MaterialGun_Eno = '` + barcode + `'`，SQL Injection 風險 | 必須使用 Dapper 參數化查詢（`@GunBarcode`） |
| Vue.js 監聽事件直接發送原始使用者輸入 | 前端不驗證直接送 API，增加後端壓力與攻擊面 | 前端先做基本格式驗證（非空、長度限制）再發送 |

### 5.2 MVP 技術底線（不可妥協）

| 技術底線 | 說明 |
|---------|------|
| 白名單讀自 `Web.config` 的 `appSettings` | 格式：`<add key="RepairPrivilegedUsers" value="02898" />`，以逗號分隔支援多人 |
| 後端 API 統一回傳格式 | `{ success: bool, message: string, data: object }` |
| 所有 SQL 使用參數化查詢 | Dapper `@param` 或 EF，絕不拼接字串 |
| Controller 層必須做後端權限驗證 | 不能只靠前端 `v-if` / `display:none` 隱藏按鈕 |
| SweetAlert2 統一錯誤呈現 | 所有失敗情境均透過 `Swal.fire` 顯示，不用 `alert()` |
| 掃碼後 Focus 回到輸入框 | 失敗後必須自動清空並重新 `focus()`，避免使用者需手動點擊 |
| 基本操作 Log | 寫入 DB 成功 / 失敗，至少要在 `Logs/` 目錄留下紀錄（現有 Log 機制） |
