## 1. 現況痛點摘要（回顧）

| 痛點 | 說明 | 本次需求關聯性 |
|------|------|--------------|
| 料槍維修手動輸入易錯 | 需目視搜尋並手動填寫/選取料槍編號，造成人為防呆破口 | **直接解決** |
| 已完修資料無特殊權限保護 | 完修後的料槍維修按鈕對所有人開放，存在誤操作風險 | **直接解決** |

## 2. 方案一：最小侵入（Minimal Touch）

### 2.1 核心概念

> 完全在現有 `MaterialGunController` 框架內，最小幅度新增 Action 與 View，延用現有 jQuery + Razor 前端技術，不引入任何新前端框架。

### 2.2 MVP 範圍

| 功能 | 實作方式 |
|------|---------|
| 新增 `ScanRepair` Action（GET + POST） | 在 `MaterialGunController` 加 2 個 Action |
| 新增 `ScanRepair.cshtml` | 複製 `MaterialGunRepair.cshtml` 改版，所有欄位初始為 `disabled` |
| 條碼輸入驗證 API | `[HttpPost] GetGunInfoByBarcode(string barcode)` 回傳 JSON |
| 前端互動 | jQuery `$.ajax()` 呼叫驗證 API，成功後 `.prop('disabled', false)` 解鎖欄位 |
| 白名單權限驗證 | Controller Action 內直接讀取 `Web.config` `appSettings["RepairPrivilegedUsers"]`，比對 Session |
| 已完修按鈕前端控制 | Razor 條件渲染，後端判斷後輸出 `disabled` 或正常 HTML |
| 主頁入口按鈕 | 在 `MaterialGunRepairView.cshtml` 工具列加一個 `<a>` 超連結按鈕 |

### 2.3 開發方式

- **複用**：現有 `ES_MaterialGunInfo`、`ES_MaterialGunRepair` EF Entity、`MaterialGunRepair.cshtml` 結構、`DepartmentAuthorizationAttribute`
- **小改**：`MaterialGunRepairView.cshtml`（加入口按鈕與按鈕權限控制）、`MaterialGunController`（加 3 個 Action）
- **新建**：`ScanRepair.cshtml`

### 2.4 優點

- 改動範圍極小，不影響其他模組
- 不需學習新技術，jQuery 即可完成
- 測試簡單，驗收快速
- 若出錯，易於回滾

### 2.5 缺點

- jQuery 直接操作 DOM，程式碼不易維護
- 欄位解鎖邏輯散落在 JS callback，維護時需追蹤多層函式
- 無法受益於 Vue.js 的響應式資料綁定，日後若需擴充欄位，需手動更新多處 JS

### 2.6 風險

| 風險 | 說明 | 機率 | 衝擊 |
|------|------|------|------|
| 前端欄位解鎖邏輯複雜 | 多個 `disabled` 欄位的聯動控制，用 jQuery 需逐一處理 | 中 | 低 |
| 未來擴充困難 | 若需增加條碼驗證欄位或步驟，jQuery 程式碼快速膨脹 | 高 | 中 |

### 2.7 開發成本估算

| 項目 | 估計 |
|------|------|
| 開發時間 | **3～4 工作天** |
| 複雜度 | **小** |
| 測試時間 | 1 天 |
| 總計 | **4～5 天** |

### 2.8 對現有系統的影響

- 改動範圍：`MaterialGunController.cs`、`MaterialGunRepairView.cshtml`、新增 `ScanRepair.cshtml`
- 不影響其他 Controller
- 資料庫無 Schema 變更

### 2.9 技術債評估

| 項目 | 評估 |
|------|------|
| 是否留下技術債 | **是** |
| 技術債內容 | jQuery 回調地獄、前端邏輯散落、無法享受 Vue 響應式帶來的可維護性優勢 |
| 未來還債成本 | **低**（日後直接替換該 View 為 Vue 版即可） |
| 建議在 MVP 中避免 | **否**（技術債可接受，換取速度） |

## 3. 方案二：Vue.js 3 CDN 強化（推薦方案）

### 3.1 核心概念

> 新增 `ScanRepair.cshtml` 時引入 Vue.js 3 CDN，以 Vue SFC-like 的方式管理頁面狀態（條碼輸入 → 資料帶出 → 欄位解鎖）。後端沿用現有 `MaterialGunController`，使用 Dapper 替代 ADO.NET 撰寫條碼驗證 API，達到前後端均提升品質，且不影響既有頁面。

### 3.2 MVP 範圍

| 功能 | 實作方式 |
|------|---------|
| 新增 `ScanRepair.cshtml` | Vue 3 CDN `createApp`，管理 `isLocked`、`gunInfo`、`repairForm` 等狀態 |
| 條碼驗證 API | 新增 `[HttpPost] GetGunByBarcode()` 使用 Dapper 查詢，回傳標準 JSON |
| 前端事件驅動 | Vue 監聽 `@keydown.enter` 觸發條碼驗證，`v-bind:disabled` 控制欄位鎖定 |
| SweetAlert2 提示 | 整合現有系統的 SweetAlert2 CDN，統一錯誤/成功提示 |
| 白名單讀取 | `ConfigurationManager.AppSettings["RepairPrivilegedUsers"]`，逗號分隔多人 |
| 後端 Action 層權限 | 在 `ScanRepairSave` Action 中直接比對 Session userId 與白名單 |
| 已完修按鈕 Razor 渲染 | Razor 判斷 `Session["Member"].fUserId` 是否在白名單，決定輸出的 HTML |
| 主頁入口按鈕 | `MaterialGunRepairView.cshtml` 工具列新增按鈕 |

### 3.3 開發方式

- **複用**：現有 EF Entity（料槍相關）、`DepartmentAuthorizationAttribute`、SweetAlert2
- **小改**：`MaterialGunController`（加 3～4 個 Action）、`MaterialGunRepairView.cshtml`
- **新建**：`ScanRepair.cshtml`（Vue 3 CDN 驅動）
- **新建**：`IMaterialGunScanService` / `MaterialGunScanService`（Dapper 查詢，可選）

### 3.4 優點

- Vue 3 響應式狀態管理，`isLocked` 一個 flag 控制所有欄位鎖定，程式碼清晰
- `@keydown.enter` 監聽掃碼行為，直覺且易讀
- 日後若需擴充欄位，只需在 Vue data 增加屬性，不需改 DOM 操作
- Dapper 查詢效能比 DataTable 好，程式碼更簡潔
- 本次 ScanRepair 頁面成為 Vue 3 CDN 的示範頁，推動全系統漸進式前端升級

### 3.5 缺點

- 開發時間比方案一多 1～2 天
- 需要開發者熟悉 Vue 3 Composition API（Options API 也可）
- 現有頁面是 jQuery，新舊頁面風格不統一（但 MVP 可接受）

### 3.6 風險

| 風險 | 說明 | 機率 | 衝擊 |
|------|------|------|------|
| Vue 3 CDN 與既有 jQuery/Bootstrap 3 衝突 | 實際上無衝突，Vue 掌管 `#app` 內部，jQuery 掌管外部 | 低 | 低 |
| 條碼輸入的 Enter 事件誤觸 | 其他欄位輸入 Enter 也可能觸發 | 中 | 低（只監聽 `txtGunBarcode` 的 Enter） |

### 3.7 開發成本估算

| 項目 | 估計 |
|------|------|
| 開發時間 | **4～6 工作天** |
| 複雜度 | **中** |
| 測試時間 | 1 天 |
| 總計 | **5～7 天** |

### 3.8 對現有系統的影響

- 改動範圍：`MaterialGunController.cs`、`MaterialGunRepairView.cshtml`、新增 `ScanRepair.cshtml`
- 可選：新增 `MaterialGunScanService.cs`（若提取 Service 層）
- `Web.config` 新增 `appSettings` 鍵值
- 資料庫無 Schema 變更

### 3.9 技術債評估

| 項目 | 評估 |
|------|------|
| 是否留下技術債 | **極少** |
| 技術債內容 | 系統其他頁面仍是 jQuery，新舊頁面前端技術不統一（可接受，計劃漸進替換） |
| 未來還債成本 | **低**（各頁面獨立，日後逐頁替換不影響其他頁面） |
| 建議在 MVP 中避免 | **白名單硬編碼**、**省略後端驗證**（兩者均透過本方案的設計避免） |

## 4. 方案三：RBAC 角色驅動（進階，不建議 MVP）

### 4.1 核心概念

> 將白名單邏輯改為透過 `ES_MemberRole` + `ES_MemberFunction` 的 RBAC 架構管控，同時建立「掃碼維修」功能的後台授權管理頁面，以動態資料庫設定取代 `appSettings`。

### 4.2 MVP 範圍（若採用）

- 全部方案二的功能
- 新增「特殊維修權限」功能代碼至 `ES_FunctionItem`
- 修改 `ES_MemberFunction` 授予特定帳號此功能
- 修改 `MaintainController` 新增功能授權管理頁面
- 重構 `DepartmentAuthorizationAttribute` 支援動態 DB 查詢

### 4.3 缺點（MVP 不適用）

- 改動範圍大，涉及授權架構全面重構
- 初期白名單只有 1 人（02898），`appsettings` 完全夠用
- 開發時間估計 **10～15 工作天**，遠超 MVP 2 週原則
- 過度設計，驗證核心需求前不應投入此成本

### 4.4 技術債評估

| 項目 | 評估 |
|------|------|
| 是否留下技術債 | **否**（一次到位） |
| 未來還債成本 | 無 |
| 建議在 MVP 中採用 | **否，MVP 後的 v1.1/v2.0 再考慮** |

## 6. 推薦方案：方案二（Vue.js 3 CDN 強化）

### 6.1 為什麼方案二最適合 MVP

1. **速度與品質的最佳平衡**：5～7 天可交付，同時建立可維護的前端架構
2. **技術債最低**：Vue 響應式狀態管理讓欄位鎖定/解鎖邏輯清晰，未來易擴充
3. **符合現有技術規劃**：全域規則已指定「新頁面可逐步引入 Vue.js 3 CDN」，本次是第一個示範頁
4. **後端無破壞性改動**：`MaterialGunController` 僅新增 Action，現有功能不受影響
5. **白名單架構合理**：`appSettings` + 逗號分隔支援多人，比方案三節省 8～10 天開發，但日後擴充成本僅是把 `appSettings` 改成 DB 查詢

### 6.2 如何避免過度設計

| 過度設計風險 | 如何避免 |
|------------|---------|
| 提前建白名單後台管理頁面 | MVP 中使用 `appSettings`，只在 README 中記錄「v1.1 再建後台」 |
| 提前重構全系統為 Vue | 只有 `ScanRepair.cshtml` 使用 Vue，其他頁面不動 |
| 提前實作 Serilog | 使用現有 `Logs/` 目錄 + `catch` 簡單記錄，Serilog 另立 Issue |
| 提前做 RBAC 重構 | 白名單暫時放 `appSettings`，RBAC 排入 v2.0 |

### 6.3 如何避免留下技術債

| 技術債底線 | 具體做法 |
|----------|---------|
| 白名單不硬編碼 | `ConfigurationManager.AppSettings["RepairPrivilegedUsers"]`，以逗號分隔 |
| 後端獨立驗證 | `ScanRepairSave` Action 必須再次比對白名單，不依賴前端 |
| 參數化 SQL | Dapper 查詢使用 `@GunBarcode`，絕不拼接字串 |
| 統一 API 格式 | 新 API 一律回傳 `{ success, message, data }` |
| 基本 Log | `try-catch` 寫入 `Logs/` 目錄 |
| 欄位初始 Disabled | 強制先掃碼才能編輯，防止繞過 |

## 7. MVP 建議實作路線圖

```
Day 1：後端基礎建設
  ├── Web.config 新增 RepairPrivilegedUsers
  ├── 新增 GetGunByBarcode API（Dapper 查詢 ES_MaterialGunInfo）
  └── 新增 ScanRepairSave Action（白名單驗證 + 寫入 ES_MaterialGunRepair）

Day 2：新增 ScanRepair.cshtml
  ├── Vue 3 CDN createApp
  ├── 條碼輸入框（自動 focus）+ @keydown.enter 監聽
  ├── v-bind:disabled 控制所有編輯欄位
  └── SweetAlert2 成功/失敗提示

Day 3：整合驗證
  ├── Ajax 呼叫 GetGunByBarcode，回填欄位
  ├── 儲存流程：POST → ScanRepairSave → 跳轉
  └── 失敗清空並重新 focus

Day 4：主頁入口按鈕 + 已完修按鈕權限
  ├── MaterialGunRepairView.cshtml 工具列加「掃碼維修」按鈕
  ├── Razor 判斷 Session userId vs 白名單，已完修按鈕條件輸出
  └── 後端 Action 同步驗證

Day 5：測試與修正
  ├── 條碼槍實測（USB HID 模式輸入 + Enter 結尾）
  ├── 各失敗情境（查無料槍、已完修、非白名單）
  └── 現有功能回歸測試（不影響 MaterialGunRepair 原頁面）
```

## 8. 技術債風險清單（本次 MVP 整體）

| 技術債 | 風險程度 | MVP 是否必須處理 | 若不處理的還債成本 |
|--------|---------|----------------|-----------------|
| 白名單存 appSettings（非 DB） | 低 | 否（可接受） | 低：日後加 DB 查詢即可，不影響邏輯 |
| jQuery 與 Vue 共存於同一系統 | 低 | 否（漸進替換） | 低：各頁面獨立，逐頁替換不影響其他頁 |
| 未引入 Serilog | 中 | 否 | 中：未來若需稽核日誌，需補加 Serilog 並補歷史操作紀錄 |
| .NET FW 4.5（非 .NET 8） | 高 | 否（超出 MVP 範圍） | 高：整個專案升級框架，工程量大 |
| EF6 Database First EDMX | 高 | 否 | 高：若要改 EF Core Code First，需重建所有 Entity |
