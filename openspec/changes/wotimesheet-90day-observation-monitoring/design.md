## Context

前個變更 `fix-wotimesheet-500-and-add-monitoring` 已完成以下工作：
- 修復 WoTimeSheet 的 SQL 參數不匹配問題（500 錯誤）
- 配置 Serilog 日誌記錄系統，已在遠端 IIS 伺服器驗證正常運作
- 日誌格式統一：`Timestamp|UserId|Department|/WoTimeSheet/FunctionName|Status|Duration|ClientIP`
- Logs 目錄已建立並具有寫入權限，能夠按日期自動輪轉日誌檔案

現狀：
- WoTimeSheet 功能每日產生日誌，記錄業務使用者的操作
- IT 部門操作已在 Serilog Filter 層被排除
- 系統準備從 2026-06-11 開始正式觀察期

## Goals / Non-Goals

**Goals:**
- 維持 90 天連續的日誌記錄（2026-06-11 至 2026-09-09），無數據遺漏
- 定期驗證日誌檔案產生狀態，及時發現異常
- 在觀察期結束後（2026-09-10），執行自動化數據分析
- 計算使用者覆蓋率：`(WoTimeSheet 不重複使用者數 / 全系統活躍業務使用者數) × 100%`
- 根據 50% 閾值結果，生成決策報告（保留/優化/廢棄功能）

**Non-Goals:**
- 不修改 WoTimeSheet 功能本身或日誌系統配置
- 不實時監控或告警（定期檢查足以）
- 不涉及新的架構、資料庫或外部依賴
- 不包括功能優化（決策後再決定）

## Decisions

### 決策 1：監控週期和檢查方式

**選項 A（選中）**：手動週檢查，每週一次查看 Logs 目錄
- **優點**：簡單低成本，便於人工確認異常；90 天週期內無需全自動
- **缺點**：依賴人工，容易遺漏

**選項 B**：自動監控指令碼（每日或每週檢查）
- **優點**：完全自動化，不易遺漏
- **缺點**：額外維護，超出觀察期的簡單性要求

**決策**：選項 A。觀察期只有 90 天，每週手動檢查 1 分鐘足以驗證日誌持續產生。

---

### 決策 2：數據分析工具

**選項 A（選中）**：Python 或 PowerShell 腳本（2026-09-10 手動執行）
- **優點**：靈活、可重複執行、易於除錯和修改
- **缺點**：需要人工執行、可能有環境依賴

**選項 B**：SQL 查詢+Excel 手工分析
- **優點**：無需編寫腳本
- **缺點**：容易出錯、難以重複

**決策**：選項 A。準備可複用的分析腳本，確保結果的準確性和可驗證性。

---

### 決策 3：覆蓋率計算方法

**邏輯**：
```
覆蓋率 = (WoTimeSheet 日誌中的不重複 UserId 計數 / 全系統活躍業務使用者的不重複 UserId 計數) × 100%

where:
  - WoTimeSheet 日誌：所有 2026-06-11 至 2026-09-09 期間的 WoTimeSheet-*.txt
  - 全系統活躍業務使用者：所有系統日誌中的不重複 UserId（排除 Department='IT'）
    或從系統用戶表查詢（非 IT 部門的活躍使用者）
```

**數據來源**：
- WoTimeSheet 使用者：從 Logs/WoTimeSheet-*.txt 提取不重複的 UserId
- 系統活躍使用者：建議從資料庫查詢或其他系統日誌統計

## Risks / Trade-offs

| Risk | 說明 | Mitigation |
|------|------|-----------|
| **日誌檔案遺漏或損壞** | 若某日伺服器宕機或磁碟滿，可能導致日誌中斷 | 每週檢查日誌檔案是否完整；若中斷，記錄受影響日期，在分析時說明 |
| **覆蓋率計算不準確** | 全系統活躍使用者數難以精確定義 | 定義清晰：使用 90 天內所有系統操作記錄的不重複 UserId（非 IT 部門）|
| **分析腳本依賴環境** | Python/PowerShell 環境可能不可用或版本不同 | 提前準備腳本，在測試環境驗證；記錄執行環境和版本 |
| **結果邊界情況** | 若覆蓋率恰好為 50%，決策不明確 | 預先定義閾值：>50% 保留，≤50% 考慮廢棄；邊界情況由管理層裁定 |

## Migration Plan

**階段 1：觀察期開始（2026-06-11）**
- 確認日誌系統正常運作
- 開始手動每週檢查日誌檔案
- 記錄檢查結果（檔案數、大小、是否產生異常）

**階段 2：持續監控（2026-06-12 至 2026-09-09）**
- 每週一次檢查 Logs 目錄
- 確認日誌檔案持續產生，無中斷
- 若發現異常，立即調查和記錄

**階段 3：數據分析（2026-09-10 起）**
- 執行分析腳本，計算覆蓋率
- 生成分析報告，含結論和建議
- 根據結果決策：保留/優化/廢棄 WoTimeSheet 功能

**回滾策略**：
- 若日誌系統故障，可暫時禁用日誌記錄，功能本身不受影響
- 觀察期中斷可延期，但應盡量避免
