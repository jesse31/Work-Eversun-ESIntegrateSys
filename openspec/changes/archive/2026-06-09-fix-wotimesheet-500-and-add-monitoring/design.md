## Context

**Current State:**
- WoTimeSheet 的 AdoNet() 和 EdList() 方法使用混亂的 SQL 參數命名（@date1, @date2, @startDate, @endDate）
- 參數陣列定義僅包含 "startDate" 和 "endDate"，導致只填起日或只填結束日時無對應參數，拋出 SqlException
- 系統無操作頻率監控，無法判斷功能使用率和必要性
- CLAUDE.md 規定使用 Serilog 記錄，Log 必須包含 Timestamp、Level、CorrelationId、Message、Exception

**Constraints:**
- 不建立新資料庫表（留待未來系統重構）
- 日誌以純文本格式存放在 `Logs/` 目錄，按日期分割
- 資料清洗規則：排除部門='IT'的開發人員操作
- 決策標準：使用者覆蓋率 < 50% 時考慮移除功能

## Goals / Non-Goals

**Goals:**
- 修復 WoTimeSheet 查詢功能的 500 錯誤
- 在 WoTimeSheet 的所有查詢和匯出功能加入結構化操作日誌記錄
- 支援 3 個月（2026-06-11 ~ 2026-09-09）的觀察期
- 日誌格式標準化，便於後續 Python/PowerShell 解析分析

**Non-Goals:**
- 不修改 API 介面或功能邏輯
- 不建立監控 Dashboard（數據分析留待 3 個月後）
- 不涉及「結束日加 1 天」邏輯的重構（留待未來優化）

## Decisions

### 決策 1：SQL 參數命名統一為 @indate/@indate2

**選項 A（選中）**：直接用 View 傳出的參數名稱 (@indate, @indate2)
- **優點**：參數名稱與 View 輸入直接對應（wono → @wono, indate → @indate），邏輯清晰，減少映射錯誤
- **缺點**：名稱不夠語義化（startDate/endDate 更具語義），但現階段修復 bug 優先

**選項 B**：統一改為 @startDate/@endDate
- **優點**：語義更清晰，支援未來的日期邏輯重構
- **缺點**：改動更大（4 處 SQL 語句），與 View 參數名稱不對應

**決策**：選項 A。符合 Karpathy 準則「最小化改動」，現階段先修復 bug，未來重構再優化參數命名。

---

### 決策 2：日誌記錄位置和時機

**選項 A（選中）**：在每個查詢/匯出方法的 finally 區塊記錄
- **優點**：確保成功和失敗都被記錄，覆蓋率更完整
- **缺點**：需在 5 個方法中重複類似邏輯

**選項 B**：建立共享的日誌工具方法
- **優點**：減少重複代碼
- **缺點**：現階段不值得抽象化，Karpathy 準則反對過度設計

**決策**：選項 A。現階段只需 5 個方法，直接記錄即可。若未來擴展到 10+ 個方法，再考慮抽象化。

---

### 決策 3：Serilog 配置方式

**選項 A（選中）**：配置 File Sink，輸出模板中包含自定義屬性 (LogContext)
```json
"outputTemplate": "{Timestamp:yyyy-MM-dd HH:mm:ss.fff}|{Properties[UserId]}|{Properties[Department]}|..."
```

- **優點**：Serilog 內建線程安全，自動處理併發寫入
- **缺點**：依賴 Serilog.Context.LogContext，需要在日誌寫入前設定屬性

**選項 B**：自己組裝字符串，直接寫入檔案
- **優點**：無額外依賴
- **缺點**：需要自己處理多執行緒 Lock，容易出錯

**決策**：選項 A。充分利用 Serilog 的成熟機制，確保日誌格式一致性和線程安全性。

---

### 決策 4：錯誤情況的日誌記錄

**標準**：記錄所有操作（成功和失敗）
- Status: "SUCCESS" 或 "ERROR"
- 失敗的操作也代表有人在用該功能，對覆蓋率計算有價值

## Risks / Trade-offs

| Risk | 說明 | Mitigation |
|------|------|-----------|
| **多執行緒並發寫入** | 高頻查詢時，多個執行緒同時寫日誌檔案 | Serilog 的 File Sink 已內建線程安全鎖機制，無需額外處理 |
| **日誌檔案無限增長** | 3 個月的日誌可能佔用大量磁碟空間 | 使用 Serilog 的日期滾動策略自動按日分割，便於後續清理 |
| **Session["Member"] 可能為 null** | 若未正常登入，無法讀取使用者資訊 | 在日誌寫入前檢查 Session["Member"]，null 時記為 "UNKNOWN" |
| **日誌內容包含分隔符號** | 若工單號碼或其他欄位包含 `\|`，破壞格式 | 在日誌寫入前，對所有欄位進行 Html/Url 編碼，或改用 JSON 格式（但違反「純文本」要求） |
| **測試期間日誌污染** | 2026-06-09 ~ 2026-06-10 測試時，測試操作會被記錄 | 測試後手動刪除 Log/WoTimeSheet-2026-06-10.txt，或在程式中加入測試環境判斷 |

---

## Migration Plan

**階段 1：修復 + 測試（2026-06-09 ~ 2026-06-10）**
- 修復 SQL 參數名稱（PR1）
- 加入日誌記錄代碼（PR2）
- 手動測試 3 個查詢場景，驗證日誌正確寫入
- 測試完成後刪除測試期間的日誌檔案

**階段 2：正式記錄（2026-06-11 ~ 2026-09-09）**
- 系統自動記錄所有 WoTimeSheet 操作
- 無需人工介入，持續 90 天

**階段 3：數據分析（2026-09-10 之後）**
- 用 Python/PowerShell 腳本清洗日誌
- 過濾掉 Department='IT' 的操作
- 計算使用者覆蓋率
- 根據覆蓋率決策是否保留功能

**回滾策略**：
- 若日誌寫入出現問題（如磁碟滿），可臨時註解掉日誌記錄代碼，系統功能不受影響
- 已寫入的日誌檔案可安全刪除，不影響系統運作
